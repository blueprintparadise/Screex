# Screex v0.6 — Region-grounded action events

**Status:** Design (approved for planning)
**Date:** 2026-06-17
**Author:** brainstormed with Claude
**Supersedes/relates:** builds on v0.5 index pipeline (`source → segment → ocr → index`).

## Summary

Screex today turns a screen recording into an ordered index of UI states, each carrying
on-screen text (`ocr_text`) and the text that appeared/disappeared since the previous state
(`text_added` / `text_removed`). An agent reading the index must *infer* what the user did
from those raw text diffs and the heuristic cursor `interactions`.

v0.6 adds a typed **action-events layer**: each state transition is classified into a
structured event — `navigate`, `type`, `click`, `open_dialog`, `error`, `scroll`, or `edit` —
**grounded to the screen region that changed**. The event is the strongest, most direct
"what did the user do" signal, sitting above `text_added` in the hierarchy of evidence.

The work is **additive and low-risk**: events are computed in a post-segmentation pass from
signals the pipeline already produces (per-line OCR boxes, text diffs, cursor hotspots,
settled keyframes). The core segmentation/OCR loop is unchanged. The feature is **opt-in**
via a `--events` flag, mirroring the existing `--boxes` / `--interactions` pattern.

The **speed** half of the original "region-diff OCR" idea (OCRing only the changed region per
frame to accelerate text-mode segmentation) is **explicitly out of scope** for v0.6 — it
touches the segmentation correctness path and will be specced separately as v0.6.1.

## Goals

- Emit a typed, region-grounded `event` per UI state, training-free and model-agnostic.
- Make the deterministic `transcript` read like real actions ("Clicked Save", "Typed … into
  API Key", "Navigated to settings", "⚠ Error: …") instead of generic "Appeared / Gone".
- Stay additive: v1 indexes still load; verbose `to_dict` output for existing fields is
  unchanged; the new field is opt-in and degrades to empty when signals are weak.
- Keep units small and independently testable, matching the existing `screex/core/` style.

## Non-goals

- No region-OCR speed optimization (deferred to v0.6.1).
- No fine-tuned UI model / ML element detection — classification is heuristic/rule-based.
- No real OS-level click/keystroke capture — events remain inferred from pixels + OCR.
- No cross-state task/narrative grouping (that was Option 3 in brainstorming; not this spec).

## Architecture

Two new bounded units under `screex/core/`, plus a post-pass in `cli.py`.

### `screex/core/regiondiff.py`

Pure numpy/cv2, no new dependencies.

```
changed_region(prev_bgr, cur_bgr) -> RegionChange | None
```

- Returns `None` when frames are absent or shapes mismatch (caller treats as "no region").
- Otherwise computes a grayscale absolute difference, thresholds it into a change mask, and
  returns a `RegionChange` dataclass:
  - `box: [x, y, w, h]` — bounding rectangle of the changed area.
  - `coverage: float` — fraction of frame area that changed (0..1).
  - `shape: str` — coarse classification of the change geometry, one of:
    - `"full"` — coverage above a high threshold (≈ whole screen replaced) → navigation-like.
    - `"overlay"` — a compact, roughly centered region while the periphery is unchanged →
      dialog/modal-like.
    - `"band"` — a tall region spanning most of the frame height with many rows differing →
      scroll-like (content shifted vertically).
    - `"local"` — a small localized region → edit/click/type-like.
- Thresholds are module-level constants with documented defaults; no new CLI knobs in v0.6.

### `screex/core/events.py`

```
classify_event(prev_state, cur_state, region, interaction, narration_text) -> dict
```

- Inputs:
  - `prev_state`, `cur_state`: `ScreenState` objects (need `ocr_text`, `text_added`,
    `text_removed`, and `boxes` — `--events` forces boxes on internally).
  - `region`: the `RegionChange | None` from `regiondiff` for this transition.
  - `interaction`: the cursor hotspot dict for `cur_state` if `--interactions` was on, else
    `None`.
  - `narration_text`: spoken text overlapping `cur_state`'s time window, if available, else
    `""` (used only as a tiebreaker / to enrich `value`, never required).
- Returns an **event dict** `{type, t, label?, field?, value?, region:[x,y,w,h], confidence}`
  or `{}` when no confident classification is possible (the state then carries no event).
- Classification rules (first match wins; each yields a `confidence` in 0..1):

  | Event | Rule |
  |-------|------|
  | `error` | any `text_added` line matches an error pattern (regex set: `error`, `failed`, `exception`, `invalid`, stack-trace markers). `label` = the matched line. |
  | `navigate` | `region.shape == "full"` **or** most of the previous text was replaced (high `text_removed` ratio). `value` = first URL-like `text_added` line if present. |
  | `open_dialog` | `region.shape == "overlay"` and prior text largely persists. `label` = a salient added line (e.g. dialog title). |
  | `scroll` | `region.shape == "band"` and added/removed sets overlap heavily (same lines, shifted). |
  | `type` | a `text_added` line sits spatially near a persistent label box (use `cur_state.boxes`), small/local region. `field` = nearest persistent label, `value` = the added line. |
  | `click` | an `interaction` exists with a small/local region. `label` = `interaction.label` (nearest text via existing `cursor.nearest_label`). |
  | `edit` | fallback for a confident-but-unclassified local change. |

- The error-pattern regex set lives in `events.py` (small, documented); it is independent of
  `redact.py` but may share style. Reusing `cursor.nearest_label` for spatial proximity keeps
  the proximity logic in one place.

### `cli.py` post-pass

After the existing segment loop builds `states` (unchanged), when `--events` is set:

1. `need_boxes` is forced `True` so every state has `boxes` (events need per-line spatial
   info). When the user did **not** also pass `--boxes`, the boxes are used for classification
   but **not** persisted into `state.boxes` (avoid surprising the output) — they are kept in a
   parallel list, same shape as the existing `state_boxes`.
2. Settled keyframes are needed for `regiondiff`. To stay memory-bounded for long recordings,
   keyframes are **re-read from disk** (`frames/NNNNN.<ext>` already written) for the prev/cur
   pair rather than held in memory.
3. Walk consecutive states `(prev, cur)`; for each, compute `region = changed_region(...)`,
   look up the cursor `interaction` and overlapping `narration`, call `classify_event`, and
   assign `cur.event` when non-empty. The first state has no predecessor → no event.

### Schema (`screex/core/index.py`)

- `ScreenState` gains `event: dict = field(default_factory=dict)`.
- `SCHEMA_VERSION` bumped `1 → 2`.
- `from_dict` reads `event` with `payload.get("event", {})` so **v1 indexes still load**
  (back-compatible; the `schema_version > SCHEMA_VERSION` guard still rejects *newer* indexes).
- `to_dict` includes `event` (via `asdict`) — empty dict for states without one.
- `compact_dict` **keeps** `event` (high-value, low-token); it continues to drop `boxes` and
  `interactions`. No new compaction logic needed beyond leaving `event` in place.

## CLI & transcript

- New flag `--events` on both `index` and `transcript` subcommands (store_true, off by
  default; documented in the README flag table). Wired through `_build`/index function
  signature alongside `boxes` / `interactions`.
- `transcript.py`: when a state has an `event`, render a typed line, e.g.:
  - `0:04 · Clicked "Save"`
  - `0:09 · Typed "sk-live-9f2a" into API Key`
  - `0:12 · Navigated to Settings`
  - `0:14 · ⚠ Error: invalid API key format`
  Falls back to the existing `**Appeared:** / **Gone:**` rendering when no event is present
  (i.e. when `--events` was not used), so default transcript output is unchanged.

## Error handling

- `changed_region` returns `None` on missing/mismatched frames; `classify_event` treats a
  `None` region as "no geometry signal" and may still classify from text/interaction, else
  returns `{}`.
- Any exception inside the events post-pass is caught per-state, appended to the index
  `warnings` (matching the existing OCR-failure diagnostic pattern), and leaves that state's
  `event` empty — it never aborts indexing.
- Mirrors how `--interactions` degrades today: best-effort, never fatal.

## Testing

- `tests/test_regiondiff.py`: synthetic BGR frame pairs asserting `box`, `coverage`, and each
  `shape` ("full" / "overlay" / "band" / "local"); `None` on shape mismatch.
- `tests/test_events.py`: hand-built `(prev_state, cur_state, region, interaction)` tuples
  asserting each event `type`, and that weak signals yield `{}`.
- `tests/test_index.py`: round-trip a `ScreenState` with an `event`; assert a v1 index (no
  `event`, `schema_version: 1`) still loads under v2.
- `tests/test_cli.py`: `--events` populates `state.event` on a tiny synthetic recording.
- `tests/test_transcript.py`: a state with an `event` renders a typed line; without one falls
  back to Appeared/Gone.
- Follows the existing one-test-file-per-module convention.

## Docs

- `README.md`: add `--events` to the `index` flag table; add `event` to the "What
  `index.json` contains" section; add a typed-transcript example.
- `SKILL.md`: teach Claude that `event` (when present) is the strongest "what did the user do"
  signal — above `text_added` — and remains approximate (inferred from pixels + OCR, no real
  click events), so trust `text_added`/`ocr_text` when they disagree.

## Rollout / versioning

- Bump package version `0.5.0 → 0.6.0` in `pyproject.toml`.
- Schema v2 is back-compatible for reads; no migration needed for existing `.screex` dirs
  (they simply have no `event` fields).

## Open questions / future (v0.6.1+)

- **Region-OCR speed optimization** (deferred): OCR only the changed region per frame to
  speed text-mode segmentation. Separate spec; must validate segmentation accuracy.
- Cross-state task/narrative grouping (brainstorming Option 3) remains a candidate.
- Parallel OCR (`--workers`) remains a candidate cheap speed win.

# Screex v0.3 — OCR Speed + Robustness Design

**Date:** 2026-06-15
**Status:** Approved (pending spec review)
**Branch:** `improve/ocr-speed` (off latest `master`)

## 0. Why (measured)

Profiling `screex index` on a real clip:
- **OCR is 93% of runtime** (54.8s of 59s); decode/resize/write/segment is ~7%.
- Each OCR call is a fixed model inference, ~insensitive to frame size or naive parallelism
  (process pools were *slower* — onnxruntime already saturates all cores).
- **onnxruntime's default threading is pathological for this small model.** Measured
  ms/frame: default ≈ **1996**, 1 thread ≈ 796, **2 threads ≈ 519**, 4 ≈ 529. So
  `intra_op_num_threads=2` is **~3.85× faster, for free.**

Two robustness gaps also surfaced:
- **No-text video → silently wrong single state.** In text mode, frames with no OCR text all
  compare "identically empty" (`text_similarity` of two empties = 1.0), so the whole
  recording collapses to one state with no warning.
- **One bad frame aborts the whole index.** An OCR exception on any frame propagates and kills
  a multi-minute run.

## 1. Goals
- ~4× faster indexing via OCR thread configuration (default, plus a `--ocr-threads` override).
- Indexing tolerates a single failing frame instead of aborting.
- No-text recordings produce a clear, actionable message instead of a silently-wrong 1 state.

## 2. Components

### 2.1 `screex/core/ocr.py`
- `_get_engine(lang=None, threads=2)`: build `RapidOCR(intra_op_num_threads=threads)`, cached
  per `(lang, threads)` key (e.g. `f"{lang or 'default'}:{threads}"`). If a RapidOCR build
  rejects the `intra_op_num_threads` kwarg (`TypeError`), fall back to `RapidOCR()`.
- `extract_text(bgr, lang=None, threads=2) -> list`:
  - `engine = _get_engine(lang, threads)` — **model-load errors propagate** (real config
    problems must be loud).
  - Wrap only the per-frame inference: `try: result, _ = engine(bgr) except Exception: return []`.
  - Otherwise unchanged (reading-order sort, strip blanks).

### 2.2 `screex/cli.py`
- `index(...)` gains `ocr_threads=2`. Pass it into OCR everywhere:
  - text mode: `segment.segment_by_text(frames, lambda b: ocr.extract_text(b, lang=lang, threads=ocr_threads), ...)`.
  - fast mode: `ocr.extract_text(bgr, lang=lang, threads=ocr_threads)`.
- New flag on the `index` subparser: `--ocr-threads` (`type=int, default=2`,
  help: "onnxruntime intra-op threads for OCR; 2 is fastest on most CPUs, 0 = library
  default"). Pass `threads=args.ocr_threads`. When `0`, treat as "library default" (call
  `RapidOCR()` with no thread kwarg) — i.e. `_get_engine` only sets the kwarg when `threads`
  is truthy.
- **No-text guard:** after states are built, if **no** state has any `ocr_text`, log (respecting
  `quiet`): `"no on-screen text detected — text mode can't split this; re-run with --fast for
  visual (motion) states."` This is detect-and-guide; it does not auto-rerun.
- `transcript` builds the index with the default `ocr_threads` (no new flag needed there).

### 2.3 Helper
A small pure helper for the no-text check (testable without OpenCV):
`_index_has_text(states) -> bool` → `any(s.ocr_text for s in states)`. The guard logs when it
returns `False`.

## 3. Data flow (unchanged except OCR construction + post-build guard)
```
frames → segment (text mode OCRs changed frames at intra_op_threads=2) → states → index.json
                                                                        ↓
                            if no state has text → log "use --fast" guidance
```

## 4. Scope

### In
- `ocr_threads` plumbing + `RapidOCR(intra_op_num_threads=…)` (default 2, `--ocr-threads` flag,
  `0` = library default, TypeError fallback).
- Per-frame inference try/except in `extract_text`.
- No-text post-build guard + `_index_has_text` helper.
- Tests; README + SKILL.md notes; bump to 0.3.0.

### Out (YAGNI / v0.4+)
- Candidate-based "OCR only settled keyframes" (fewer calls) — bigger structural change.
- Parallel OCR worker pool (measured *slower* on short clips; uncertain win).
- Rotation-metadata handling for phone recordings.
- GPU / alternate OCR backends.

## 5. Testing & success criteria
- **Thread param plumbs:** `extract_text(img, threads=2)` reads known rendered text (extends the
  existing OCR test or adds a `threads=2` assertion); `_get_engine` caches distinct
  `(lang, threads)` keys without error.
- **Per-frame tolerance:** monkeypatch `ocr._get_engine` to return an engine whose `__call__`
  raises; `extract_text(<any frame>)` returns `[]` (does not raise).
- **No-text guard:** `_index_has_text([])` and an all-empty-`ocr_text` state list → `False`; a
  state with text → `True`. (CLI-level: optionally assert the message is logged on a
  text-free synthetic clip.)
- **Speed (manual):** re-profile the same clip; OCR ms/frame drops from ~2000 to ~520
  (~3.85×). Record before/after in `docs/eval.md`.
- Full suite + ruff stay green.

## 6. Environment / process
Windows 11, Python, opencv + numpy + rapidocr-onnxruntime. `python -m pytest` and
